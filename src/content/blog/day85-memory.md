<p class="lead">LLM 本身<strong>没有记忆</strong>——每次调用都是无状态的，它「记得」的一切，本质上都是这一轮被塞进上下文窗口的文本。所谓 Agent 的记忆机制，就是<strong>用工程手段，决定每一轮该把什么放进这个窗口</strong>。本篇从最基本的问题起步：为什么需要记忆，记忆又分哪几种。</p>

## Part 1：为什么需要记忆，记忆有哪几种

### 1.1 为什么需要记忆：模型是无状态的

先把最容易被忽略的事实摆正：**模型权重在推理时不会更新，两次 API 调用之间它什么都不记得。**

<div class="cycle">
<span class="cycle-node">每轮请求</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">把历史 + 现状拼成上下文</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">模型一次性读完、输出</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">用完即忘</span>
<span class="cycle-note">「记忆」不在模型里，在你每轮重新拼出来的那段文本里</span>
</div>

所以记忆机制要回答的从来不是「模型怎么记住」，而是：**这一轮，我该从茫茫历史里捞出哪些信息，拼进有限的上下文窗口。** 由此自然分成两层——当前任务用的工作记忆，和跨任务沉淀的长期记忆。

### 1.2 记忆有哪几种

按「生命周期」切一刀，记忆分成两层：撑住当前任务的**工作记忆（短期）**，和跨会话沉淀的**长期记忆**。

| | **工作记忆（短期）** | **长期记忆** |
|---|---|---|
| 载体 | 上下文窗口本身（当轮 prompt） | 外部存储：向量库 / DB / 文件 / KG |
| 生命周期 | 单次会话 / 单个任务，结束即弃 | 跨会话、跨任务持久保存 |
| 容量 | 受窗口 token 数硬限制 | 近乎无限 |
| 访问方式 | 直接「在场」，模型全程可见 | 需检索后注入，不命中就用不上 |
| 速度 | 即时 | 多一跳检索（latency / 可能漏召回） |
| 典型内容 | 当前对话历史、刚才的 observation、临时草稿 | 用户画像、历史偏好、知识库、过往经验教训 |

**为什么需要工作记忆**：多步任务里，第 5 步要用到第 2 步的 observation、第 3 步定的计划——没有工作记忆，每一步都成了失忆的重新开始，ReAct / Plan-and-Execute 的闭环根本转不起来。

**为什么需要长期记忆**：工作记忆会随会话结束清空，且受窗口大小限制。用户上周说过「我对花生过敏」、系统三个月前踩过的坑——这些不可能也不该常驻上下文，必须沉到外部存储，用到时再捞回来。一句话：<strong>工作记忆解决「这次任务连不连贯」，长期记忆解决「跨次之间长不长记性」。</strong>

<aside class="callout">
<p><strong>💡 类比</strong>：工作记忆 ≈ 人的「桌面」，地方小、手边即用、下班清空；长期记忆 ≈ 「档案柜」，容量大、要先去翻、长期保存。Agent 的活，就是不断在档案柜和桌面之间搬运正确的纸。</p>
</aside>

至此，「为什么要记」和「记忆分几种」就清楚了。接下来的问题才是工程重头：工作记忆装不下怎么办、长期记忆怎么读写与遗忘——留到后面几个 Part 展开。

## Part 2：工作记忆的处理方法

### 2.1 一个矛盾催生所有方法

工作记忆的所有处理方法，都在解同一道题：

<div class="cycle">
<span class="cycle-node">任务越长 → 历史越堆越多</span>
<span class="cycle-arrow">vs</span>
<span class="cycle-node">上下文窗口是死的（token 有上限）</span>
<span class="cycle-note">还要和 system prompt、工具 schema、检索结果、输出一起抢这点空间</span>
</div>

朴素做法是「全量塞」——把每一轮历史原样拼进 prompt。任务一长必然撞墙：**要么超出窗口直接报错，要么虽然没超但被无关历史稀释、触发 lost-in-the-middle，钱还越花越多。** 于是有了下面几种方法，各自砍向这道题的不同侧面。

### 2.2 四种主流方法，分别解决什么

| 方法 | 怎么做 | 解决了什么问题 | 代价 / 副作用 |
|---|---|---|---|
| **全量拼接** | 历史原样全塞进 prompt | 信息零丢失、实现最简单 | 会超窗、被稀释、成本高——只适合短任务 |
| **滑动窗口** | 只留最近 N 轮，更早的丢弃 | 直接控住长度，防超窗、防稀释 | 硬性遗忘早期信息，窗外的一律拿不到 |
| **摘要压缩** | 把较早历史用模型压成摘要，替换原文 | 在有限 token 里保住早期主干，比滑窗少丢信息 | 摘要有损，反复压会失真累积、增加一次 LLM 调用 |
| **检索式记忆** | 历史存外部库，每轮按当前 query 只捞相关片段注入 | 让长度和历史总量<strong>解耦</strong>，任务再长窗口也不涨 | 依赖召回质量，漏召回=用不上、多一跳延迟 |

四者不是替代而是**递进**——每一种都在补上一种的短板：

<div class="cycle">
<span class="cycle-node">全量：会超窗</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">滑窗：砍长度，但硬遗忘</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">摘要：把丢掉的压成主干留住</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">检索：需要哪段捞哪段，长度与总量解耦</span>
</div>

### 2.3 还有一类：把状态卸载到外部

前面四种处理的是「对话历史」这类<strong>非结构化文本</strong>。还有一类信息更适合结构化管理——当前计划、待办清单、中间变量、草稿：

<div class="gridcards">
<div class="gridcard"><b><span class="gc-ico">🗂️</span>结构化卸载</b>把状态写进外部 scratchpad / 文件 / 变量表，prompt 里只留引用或摘要，需要时再读回。解决「过程状态占满窗口」的问题。</div>
<div class="gridcard"><b><span class="gc-ico">📌</span>钉住关键信息</b>把 system prompt、当前目标、硬约束单独固定在窗口头/尾，不参与滑窗和压缩。解决「压缩误删关键约束」的问题。</div>
</div>

这类方法与前面正交：前者管「历史文本怎么装下」，它管「过程状态放哪、关键信息别弄丢」。

### 2.4 实战：混合才是常态

真实 Agent 很少只用一种，而是按「信息离当前有多近」分层处理：

<div class="cycle">
<span class="cycle-node">最近 N 轮：原文保真</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">较早历史：压成摘要</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">更早 / 海量：进库按需检索</span>
<span class="cycle-note">再叠上：关键约束钉死、过程状态卸载到外部</span>
</div>

一句话记住这套分层原则：<strong>最近的留原文、较远的留摘要、相关的可召回、关键的钉死。</strong>

<aside class="callout">
<p><strong>🧭 一句话</strong>：工作记忆的处理，本质是在「装得下」和「不丢关键信息」之间做<strong>有损取舍</strong>。选方法前先问一句——我到底在解哪个问题：是超窗（砍长度），是被稀释（提精度），还是历史太多装不下（解耦）？问题对上了，方法自然就选对了。至于「压缩会失真、检索会漏召回」这些具体的坑怎么防，下一 Part 细说。</p>
</aside>

## Part 3：工作记忆怎么 evaluate

### 3.1 为什么要单独评估它

Part 2 的每种方法都是<strong>有损</strong>的——滑窗会丢、摘要会歪、检索会漏。问题是：这些「损」到底伤没伤到任务？不评估，你根本不知道是记忆策略在拖后腿，还是模型本身不行。

而且工作记忆的坑很隐蔽：**它几乎不会直接报错，而是让任务「悄悄变差」。** 模型忘了三轮前定的约束、把被压缩失真的信息当真、复述了窗外早已推翻的结论——表面还在正常输出，实则地基已歪。所以要专门给它设一套评估。

### 3.2 一个前提：分清是「记忆的错」还是「模型的错」

评估工作记忆，最容易踩的坑是把两类失败混为一谈：

<div class="cycle">
<span class="cycle-node">该有的信息不在窗口里</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">记忆策略的锅（丢了/压歪了/没召回）</span>
</div>

<div class="cycle">
<span class="cycle-node">信息在窗口里，模型却用错了</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">模型能力的锅（没读懂/被中段忽略）</span>
</div>

区分方法很朴素：**把「理想上下文」（人工挑出的、恰好够用的历史）喂给模型跑一遍**，作为上界基线。若理想上下文能做对、而你的记忆策略做错了，差距就是记忆策略造成的损失；若理想上下文也做不对，那是模型或任务本身的问题，别怪记忆。

### 3.3 两层指标：过程层 + 结果层

评估分两层看——记忆策略「装得对不对」，以及它最终「帮没帮上任务」。

<strong>① 过程层（直接量记忆质量，可离线算）</strong>

| 指标 | 衡量什么 | 怎么理解 |
|---|---|---|
| **召回率 / 覆盖率** | 该保留的关键信息，有多少真的还在窗口里 | 漏掉一条硬约束，召回就扣分——直指「丢没丢」 |
| **精度 / 信噪比** | 窗口里的内容有多少是当前真正相关的 | 越低说明无关历史越多，越容易触发稀释 |
| **压缩保真度** | 摘要 vs 原文，关键事实有没有走样 | 抓摘要有损、失真累积的问题 |
| **token 效率** | 达到同等效果用了多少 token | 同样做对，谁更省——成本维度 |

<strong>② 结果层（看对任务的实际增益，端到端）</strong>

| 指标 | 衡量什么 |
|---|---|
| **任务成功率** | 换用某记忆策略后，端到端把任务做对的比例——最终裁判 |
| **多轮一致性** | 后面几轮有没有和前面自相矛盾、忘记已定的约束 |
| **长程依赖命中** | 需要用到很早之前信息的那些步，答对了没有 |
| **成本 / 延迟** | 达到同等成功率的 token 花费与耗时 |

<aside class="callout">
<p><strong>💡 别只看过程层</strong>：召回率高不等于任务就好——召回全塞进去可能反而稀释、超窗。过程层帮你定位问题，<strong>结果层（尤其任务成功率）才是最终标准</strong>。两层要一起看。</p>
</aside>

### 3.4 怎么落地评估

<div class="gridcards">
<div class="gridcard"><b><span class="gc-ico">🎯</span>造长程测试集</b>专门设计「答案埋在很早之前」的多轮任务——只有真的记住早期信息才能答对，直击工作记忆软肋。</div>
<div class="gridcard"><b><span class="gc-ico">💉</span>探针 / 针在草堆</b>往历史里塞一条关键事实（needle），末尾提问考它——直接测「这条还在不在、找不找得到」。</div>
<div class="gridcard"><b><span class="gc-ico">📍</span>位置扫描</b>把关键信息放在窗口的头/中/尾分别测，量化 <strong>lost-in-the-middle</strong> 对你这套策略的影响。</div>
<div class="gridcard"><b><span class="gc-ico">⚖️</span>消融对比</b>同一批任务下，全量/滑窗/摘要/检索各跑一遍，对比成功率与成本，用数据选型而非拍脑袋。</div>
</div>

<aside class="callout">
<p><strong>🧭 一句话</strong>：评估工作记忆，先用「理想上下文」把<strong>记忆的锅和模型的锅分开</strong>，再用过程层（召回/精度/保真/token）定位问题、用结果层（任务成功率/一致性/长程命中）下最终结论。核心就一句——<strong>你砍掉的历史，到底伤没伤到任务？</strong></p>
</aside>
