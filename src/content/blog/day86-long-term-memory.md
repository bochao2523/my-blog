<p class="lead">上一篇聊过，工作记忆解决的是「这次任务连不连贯」，而长期记忆解决的是<strong>「跨次之间长不长记性」</strong>。本篇聚焦长期记忆本身：它为什么重要，又该怎么维护——写入、更新、遗忘与一致性。</p>

## Part 1：长期记忆为什么重要

没有长期记忆，Agent 每次会话都是一张白纸：上一次踩过的坑、用户提过的偏好、系统攒下的经验，全部随会话结束一起清空。

<div class="cycle">
<span class="cycle-node">没有长期记忆</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">每轮从零开始</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">重复问一样的问题、重复犯一样的错</span>
<span class="cycle-note">用户的耐心和信任，就是在这种「失忆」里被慢慢耗光的</span>
</div>

长期记忆真正撑起的是三件事：**个性化**（记住用户是谁、偏好什么，而不是每次重新自我介绍）、**积累**（把一次调试、一次失败沉淀成经验，下次直接复用而不是重蹈覆辙）、**一致性**（跨会话不自相矛盾，昨天说过的话今天还认）。

<aside class="callout">
<p><strong>🧭 一句话</strong>：工作记忆决定 Agent 能不能<strong>完成</strong>一件事，长期记忆决定它能不能<strong>越用越懂你</strong>。没有前者，任务做不完；没有后者，Agent 永远是个初次见面的陌生人。</p>
</aside>

## Part 2：长期记忆的储存方式与调用方式

长期记忆要解决两件独立的事：**存进去**（写入之后放在哪、以什么形态存）和**取出来**（下次要用时，怎么找到它、怎么塞回上下文）。二者要分开看——存储方式决定了「有没有留下」，调用方式决定了「用得上用不上」。

### 2.1 储存方式：按「存的是什么」分三类

抛开具体数据库选型，先按内容性质分类，这是决定该用什么存储技术的根源：

| 类型 | 存的是什么 | 典型例子 | 常见载体 |
|---|---|---|---|
| **语义记忆**（Semantic） | 与具体某次经历无关的事实性知识 | 用户画像、领域知识、产品文档 | 向量库（语义检索）、结构化 DB（精确字段） |
| **情景记忆**（Episodic） | 具体发生过的某一次事件 | 某次对话摘要、某次任务的执行轨迹与结果 | 向量库 + 时间戳、日志 / 文档存储 |
| **程序记忆**（Procedural） | 怎么把一件事做好的方法本身 | 验证有效的工作流、system prompt 的迭代版本 | 代码 / 配置文件、prompt 模板库 |

<div class="cycle">
<span class="cycle-node">语义记忆：我知道什么</span>
<span class="cycle-arrow">+</span>
<span class="cycle-node">情景记忆：我经历过什么</span>
<span class="cycle-arrow">+</span>
<span class="cycle-node">程序记忆：我该怎么做</span>
<span class="cycle-note">三者共同构成一个「越用越老练」的 Agent</span>
</div>

再看落地时的具体存储技术，四种各有侧重，往往不是单选而是组合使用：

| 存储技术 | 擅长存 / 查 | 优点 | 局限 |
|---|---|---|---|
| **向量库** | 非结构化文本，按「语义相近」检索 | 能捞到字面不同但意思相关的内容 | 不擅长精确过滤、多跳关系推理 |
| **结构化 DB / KV** | 有固定字段的事实（画像、配置、状态） | 精确查询、易更新、易做约束校验 | 存不下模糊的、非结构化的长文本 |
| **文件 / 文档存储** | 原始日志、长文档、历史记录全量归档 | 保真、便宜、易追溯 | 不能直接语义检索，通常要配合索引 |
| **知识图谱（KG）** | 实体与实体之间的关系 | 支持多跳推理（A 关联 B 关联 C） | 构建与维护成本高，schema 设计门槛高 |

<aside class="callout">
<p><strong>💡 怎么选</strong>：问自己「查的时候是靠意思相近，还是靠精确字段，还是靠关系链」。三个答案分别指向向量库、结构化 DB、知识图谱；文件存储则永远是那份「保真备份」，其他存储都可以看作是它之上建的索引。</p>
</aside>

### 2.2 调用方式：怎么把存进去的东西再捞出来

存储只完成一半，另一半是「这一轮该不该查、查什么、查回来的东西怎么用」。可以拆成三层问题：

**① 什么时候触发检索**

<div class="gridcards">
<div class="gridcard"><b><span class="gc-ico">🔄</span>自动注入</b>每轮开始前，系统自动用当前输入去检索一次，结果无条件拼进 prompt。简单可靠，但可能查了用不上、也可能该查的时候没查。</div>
<div class="gridcard"><b><span class="gc-ico">🛠️</span>工具调用式</b>把检索包装成一个工具（如 <code>search_memory</code>），模型自己判断要不要调、查什么关键词。更精准，但依赖模型「知道自己不知道」。</div>
</div>

**② 用什么算法找到相关的那一条**

| 算法 | 怎么排序 / 匹配 | 适合场景 |
|---|---|---|
| **向量相似度** | embedding 距离最近的 top-k | 模糊语义匹配，「意思像」就行 |
| **关键词 / BM25** | 字面词频匹配 | 需要精确命中专有名词、代号 |
| **混合检索 + 重排** | 先粗筛（向量或关键词）再用 rerank 模型精排 | 兼顾召回面和精度，工程上最常见 |
| **时间衰减加权** | 越新的记录权重越高 | 用户偏好等「会过期」的信息 |
| **重要性 / 频率加权** | 被标记重要或反复用到的记录优先 | 区分「说过一次」和「反复强调」 |
| **图遍历（多跳）** | 沿知识图谱的关系边扩展 | 需要「A 的同事的项目」这类链式查询 |

**③ 捞回来的东西怎么塞进上下文**

<div class="cycle">
<span class="cycle-node">检索出一批候选</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">按相关性 / 时间 / 重要性排序、去重</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">超长的先摘要，太多的只取 top-k</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">拼进 prompt 的固定位置（如 system prompt 附近）</span>
</div>

<aside class="callout">
<p><strong>🧭 一句话</strong>：储存方式回答「记忆该长成什么样、放在哪」，调用方式回答「什么时候查、怎么排序、怎么塞回去」。两边任何一边偷工减料，都会导致同一个结果——<strong>记住了，但用不上</strong>。至于记忆本身怎么被写入、更新、过期淘汰，留给后面的 Part 展开。</p>
</aside>

## Part 3：长期记忆的写入、更新、遗忘机制

长期记忆不是一张只会往下追加的日志表。真正的维护工作发生在三个动作里：**写入**（什么值得被记住）、**更新**（记住的东西错了或过时了怎么办）、**遗忘**（什么时候该主动扔掉）。三者共同决定了记忆库是「越用越聪明」还是「越用越臃肿混乱」。

### 3.1 写入：不是所有对话都值得被记住

最朴素的做法是把原始对话整段存进去，但这会让记忆库迅速塞满大量低价值的噪声。工程上通常在写入前插一道**提炼**环节：

<div class="cycle">
<span class="cycle-node">原始交互（对话 / 执行轨迹）</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">LLM 提炼出「值得记住的事实」</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">打分：重不重要、要不要存</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">与已有记忆比对去重后，才真正写入</span>
</div>

几个关键设计点：

| 环节 | 要解决的问题 | 常见做法 |
|---|---|---|
| **提炼** | 原文太啰嗦、太口语化，不适合直接存 | 用一次 LLM 调用把对话压成原子化的事实条目（一条记忆只讲一件事） |
| **打分** | 不是每条事实都同等重要 | 让模型给「重要性」打分（如 1–10），低分的不存或只做低优先级存档 |
| **去重 / 合并** | 同一件事反复被提及，写多次会冗余 | 写入前检索一次相似记忆，命中则合并 / 更新而非新增 |
| **时机** | 什么时候触发写入 | 实时（每轮结束就判断）或离线（会话结束后批量提炼），后者更省调用但有延迟 |

<aside class="callout">
<p><strong>💡 别把「记录」和「记住」混为一谈</strong>：记录是保真的原始日志，写进文件存档即可；记住是提炼后的、真正会被检索调用的记忆条目。前者追求不丢信息，后者追求「少而准」。</p>
</aside>

### 3.2 更新：记忆需要能被「纠正」，不能只增不改

用户的偏好会变，之前记下的信息也可能是错的或过时的——如果只支持追加，记忆库里会同时存在「用户喜欢吃辣」和「用户现在不吃辣了」两条互相矛盾的记录，检索时到底信哪条？

<div class="cycle">
<span class="cycle-node">新信息进来</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">先检索是否已有相关旧记忆</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">判断关系：无关 / 补充 / 冲突</span>
<span class="cycle-arrow">→</span>
<span class="cycle-node">分别处理：新增 / 合并 / 覆盖旧记录</span>
</div>

| 更新策略 | 怎么做 | 适合场景 |
|---|---|---|
| **仅追加（append-only）** | 新记忆永远新增，旧记忆保留不动 | 需要完整历史审计，或情景记忆本身就该按时间线保存 |
| **覆盖更新（in-place）** | 检测到冲突，用新值直接替换旧值 | 用户画像、偏好一类「此刻只有一个真值」的字段 |
| **版本化（versioning）** | 保留旧值但标记为「已过期」，新值置顶 | 既要知道现状，又想保留「曾经是什么」的轨迹 |

冲突检测本身也依赖检索——写入前先把新信息去查一遍相关旧记忆，靠语义相似度或结构化字段匹配判断是不是在说同一件事。

### 3.3 遗忘：主动扔掉，而不是任由堆积

遗忘常被误解成「缺陷」，但对长期记忆而言它是<strong>必需的维护动作</strong>：记忆无限累积，检索时噪声越来越多、召回精度下降，存储和检索成本也在涨，过时信息还可能反过来误导模型。

<div class="gridcards">
<div class="gridcard"><b><span class="gc-ico">⏳</span>时间衰减</b>越久没被用到的记忆，检索时权重越低——类比人类的遗忘曲线，让「新鲜」的信息天然更容易被捞到。</div>
<div class="gridcard"><b><span class="gc-ico">📉</span>低频淘汰</b>长期不被访问、也从未被验证有用的记忆，定期清理或降级存档，类似缓存里的 LRU。</div>
<div class="gridcard"><b><span class="gc-ico">🗜️</span>压缩合并</b>不是硬删除，而是把大量低价值细节压缩成一条摘要——细节丢了，但主干和结论留住了。</div>
<div class="gridcard"><b><span class="gc-ico">🎯</span>显式过期 / 主动删除</b>给会过期的信息设 TTL（如「本季度优惠信息」），或响应用户「忘掉刚才那句」的主动删除请求。</div>
</div>

<aside class="callout">
<p><strong>🧭 一句话</strong>：写入决定「进不进记忆库」，更新决定「里面的东西对不对」，遗忘决定「留多久、该不该让位」。三者其实是同一套维护逻辑在不同阶段的体现——<strong>没有筛选的写入，没有纠错的更新，没有节制的存储，最终都会喂出一个越用越不可信的记忆库。</strong></p>
</aside>

## Part 4：长期记忆怎么评估

### 4.1 为什么要单独评估

长期记忆的问题很少在写入当下暴露——它一般会<strong>潜伏</strong>：记错的偏好、该更新没更新的旧信息、该删没删的隐私数据，往往要等几天甚至几周后的某次检索才会突然「翻车」。不专门去测，就只能靠用户抱怨来发现问题，而那时往往已经造成了体验损失。

### 4.2 按生命周期拆开测：写入、更新、遗忘各自的指标

Part 3 把长期记忆的维护拆成写入、更新、遗忘三个动作，评估也照着这条线走——每个阶段问一个不同的问题：

| 阶段 | 核心问题 | 具体指标 |
|---|---|---|
| **写入** | 该记的记住了吗？记的准不准？ | 关键事实提炼召回率、重要性打分与人工标注的一致度、同一事实重复写入的冗余率 |
| **更新** | 冲突有没有被发现，处理对不对？ | 冲突检测召回率（该发现的矛盾有没有漏）、更新后正确率（新值是否真的覆盖了旧值） |
| **遗忘** | 该忘的忘了吗？不该忘的还留着吗？ | 过期信息残留率（越低越好）、误删率（不该删却删了，越低越好） |
| **调用** | 存在库里的东西，跨会话还能稳定捞出来吗？ | 跨会话一致性、长期偏好 / 硬约束的命中率 |

<aside class="callout">
<p><strong>💡 更新和遗忘的指标要成对看</strong>：只看「该删的删了没」容易矫枉过正——把还有用的信息也一并清掉。所以每项都要配一个反向指标（漏了多少 / 错删了多少），单独看一个数字很容易被误导。</p>
</aside>

### 4.3 结果层：记忆是不是真的让 Agent「越用越好」

过程层指标定位问题出在哪一步，但最终要看端到端效果：

| 指标 | 衡量什么 |
|---|---|
| **个性化任务成功率** | 需要用到用户历史信息才能做对的任务，做对了多少 |
| **跨会话一致性** | 多次会话之间，Agent 有没有自相矛盾、说了又忘 |
| **随时间的稳定性** | 记忆库用得越久，效果是在变好还是在腐化（噪声堆积、检索精度下滑） |
| **纠错响应率** | 用户纠正过的信息，之后还会不会被错误地翻出来用 |

### 4.4 怎么落地评估

<div class="gridcards">
<div class="gridcard"><b><span class="gc-ico">📅</span>跨会话回归测试</b>模拟多轮跨天 / 跨会话的对话脚本，检查后续会话是否正确记住前面提到的信息，不需要用户重复自己。</div>
<div class="gridcard"><b><span class="gc-ico">⚡</span>矛盾注入测试</b>故意在后续会话给出与旧记忆冲突的新信息，检验系统是否正确覆盖更新，而不是新旧记录同时存在、检索时随机命中。</div>
<div class="gridcard"><b><span class="gc-ico">🗑️</span>过期 / 删除合规测试</b>设置 TTL 或发出「忘掉这条」的请求，过后专门验证它是否真的从检索结果里消失。</div>
<div class="gridcard"><b><span class="gc-ico">📈</span>长期漂移监控</b>线上定期抽样检查记忆库的信噪比、冗余率、平均记忆年龄等趋势指标，及早发现「越用越乱」的苗头。</div>
</div>

<aside class="callout">
<p><strong>🧭 一句话</strong>：评估长期记忆，本质是在问三个问题——<strong>该记的有没有记住，该改的有没有改对，该忘的有没有忘掉</strong>。过程层指标帮你定位是写入、更新还是遗忘的问题，结果层（个性化成功率、跨会话一致性、随时间的稳定性）才是最终检验：记忆库用得越久，Agent 应该越懂你，而不是越糊涂。</p>
</aside>
